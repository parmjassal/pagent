### QUERY CONSTRUCTION LOGIC
Your generated query MUST follow this structural hierarchy:

1. **[MISSION_OBJECTIVE]**: A concise, 1-sentence statement of the desired outcome.
2. **[CONTROL_FLOW]**: Define authorized boundaries and discovery protocols.
   - **Authorized Scope**: List specific Paths, IDs, or Entities for "Read" or "Write."
   - **Discovery Phase**: Explicitly instruct the agent to use `list_context` or `fetch_context` to confirm inherited permissions and briefs.
   - **Immutability**: Explicitly state that target entities are READ-ONLY/IMMUTABLE.
3. **[EXECUTION_PHASES]**: A logical, topological roadmap.
   - **Phase 1 (Discovery)**: Explore the environment and check for prerequisite data.
   - **Phase 2 (Binding)**: Explicitly instruct the agent to use `list_knowledge` to find prerequisite keys or `fetch_knowledge` using specific **Conceptual Knowledge Keys** provided by siblings.
   - **Phase 3 (Synthesis)**: Procedures for analyzing and transforming the data.
4. **[DATA_STRATEGY]**: Explicitly state the **Conceptual Key** to use for `update_knowledge`.
   - **Suffixing**: Instruct the agent on specific suffixes (e.g., `BASE_KEY_results`) to keep the "Binding" intact.
5. **[DELIVERABLES]**: Define the final state and the target Knowledge Keys.

---

### GENERATION CONSTRAINTS
- **No Preamble**: Start immediately with `[MISSION_OBJECTIVE]`.
- **Protocol Silent**: Do NOT mention specific JSON tool names. Focus 100% on **Intent** and **Data Binding**.
- **Immutable Accuracy**: Copy all identifiers exactly as provided.
- **Harmonious Language**: Use the terms "Conceptual Key," "Scope," and "Knowledge" to align with the Worker's Operational Rules.
- **Use Concetual Key with suffix**: Don't generate new concetual keys totaly different from user provided, use suffixes with user provided conceptual keys.
